Как построить DDD-модуль на Rails без единой внешней зависимости

Разбираем на реальном примере — модуле управления подписками. От Value Object до Aggregate Root, репозитория и доменных событий. Только то, что уже есть в Rails и Ruby.

Мотивация

Зачем вообще DDD, если есть ActiveRecord

Стандартный Rails-подход — «толстая модель» (fat model), где ApplicationRecord совмещает персистентность, бизнес-правила, валидации, колбэки и связи. На небольшом приложении это работает прекрасно. Боль начинается, когда бизнес-логика разрастается: колбэки дёргают друг друга, валидации зависят от контекста, а один класс отвечает сразу за всё.

DDD предлагает разделить модель предметной области (как устроен бизнес) и модель хранения (как лежит в базе). Доменные объекты ничего не знают про SQL, колбэки и Rails — это просто Ruby-объекты, выражающие правила. ActiveRecord становится деталью инфраструктуры, спрятанной за репозиторием.

Ключевая идея
Домен — чистый Ruby. Rails — инфраструктура. Граница между ними проходит по репозиторию и публичному API модуля. Внутрь домена ActiveRecord не протекает.

Возьмём конкретную задачу: модуль подписок. Клиент оформляет платную подписку на тариф, она активна определённый период, её можно поставить на паузу, возобновить, отменить. При отмене раньше срока — частичный возврат. Это классический ограниченный контекст (bounded context) с богатыми правилами.

Структура модуля внутри Rails-приложения

Модуль живёт под app/modules/subscriptions и изолирован от остального приложения. Внутри — собственная слоистая структура. Имя верхнего namespace Subscriptions повторяет ограниченный контекст.

# app/modules/subscriptions/ subscriptions/ ├─ domain/ # чистый Ruby, ноль Rails │ ├─ value_objects/ │ │ ├─ money.rb │ │ ├─ billing_period.rb │ │ └─ subscription_status.rb │ ├─ entities/ │ │ └─ subscription.rb # Aggregate Root │ ├─ events/ │ │ ├─ subscription_activated.rb │ │ ├─ subscription_paused.rb │ │ └─ subscription_cancelled.rb │ └─ repositories/ │ └─ subscription_repository.rb # интерфейс ├─ application/ # оркестрация use-cases │ ├─ activate_subscription.rb │ ├─ cancel_subscription.rb │ └─ pause_subscription.rb ├─ infrastructure/ # здесь живёт ActiveRecord │ ├─ records/ │ │ └─ subscription_record.rb # ApplicationRecord │ └─ ar_subscription_repository.rb # реализация └─ subscriptions.rb # фасад — публичный API

Чтобы Rails автоматически загружал классы, добавьте путь в автозагрузку (Zeitwerk сам разберёт namespace по папкам):

config/application.rb
module YourApp
  class Application < Rails::Application
    # подключаем модули как обычные autoload-пути
    config.autoload_paths += Dir[Rails.root.join("app/modules")]
    config.eager_load_paths += Dir[Rails.root.join("app/modules")]
  end
end

Тактический паттерн · 01

Value Objects — значения без идентичности

Value Object описывается своими атрибутами, а не идентификатором. Два объекта Money(100, "EUR") равны и взаимозаменяемы. Они неизменяемы (immutable) и содержат правила, относящиеся к самому значению. В Ruby их удобно строить на Data.define (доступен с Ruby 3.2) — он даёт неизменяемость и равенство по значению из коробки.

domain/value_objects/money.rb
module Subscriptions
  module Domain
    # Data.define → immutable, == по значению, без идентичности
    Money = Data.define(:amount_cents, :currency) do
      def initialize(amount_cents:, currency: "EUR")
        raise ArgumentError, "amount must be integer cents" unless amount_cents.is_a?(Integer)
        raise ArgumentError, "negative money" if amount_cents.negative?
        super
      end

      def +(other)
        ensure_same_currency!(other)
        with(amount_cents: amount_cents + other.amount_cents)
      end

      def *(factor)
        with(amount_cents: (amount_cents * factor).round)
      end

      def zero? = amount_cents.zero?

      def to_s = format("%.2f %s", amount_cents / 100.0, currency)

      private

      def ensure_same_currency!(other)
        raise ArgumentError, "currency mismatch" if currency != other.currency
      end
    end
  end
end

Второй Value Object — BillingPeriod. Он инкапсулирует правило «период длиной в месяц/год начинается тогда-то» и умеет вычислять остаток дней — это пригодится при возврате.

domain/value_objects/billing_period.rb
module Subscriptions
  module Domain
    BillingPeriod = Data.define(:starts_on, :ends_on) do
      UNITS = { monthly: 1, yearly: 12 }.freeze

      def self.starting(on:, unit:)
        months = UNITS.fetch(unit) { raise ArgumentError, "unknown unit" }
        new(starts_on: on, ends_on: on.next_month(months) - 1)
      end

      def covers?(date) = (starts_on..ends_on).cover?(date)

      def total_days = (ends_on - starts_on).to_i + 1

      def remaining_days_from(date)
        return 0 unless covers?(date)
        (ends_on - date).to_i
      end
    end
  end
end
Антипаттерн
Хранить деньги как float в колонке и складывать вручную в контроллере. Округления поплывут, а правило «нельзя складывать евро с долларами» нигде не зафиксировано. Value Object делает невалидное состояние непредставимым.
Тактический паттерн · 02

Entities — объекты с идентичностью

Entity, в отличие от Value Object, имеет идентификатор и живёт во времени: его атрибуты меняются, но это всё тот же объект. Подписка с id=42 остаётся той же подпиской после паузы и возобновления. В нашем модуле главная сущность — Subscription, и она же является корнем агрегата, поэтому полный её код — в следующем разделе.

Важный момент: доменная Entity — это не ActiveRecord-модель. Это обычный Ruby-класс. Идентичность задаётся через id, равенство — по типу и идентификатору, а не по всем полям:

domain/entities/base_entity.rb (опционально)
module Subscriptions
  module Domain
    module Entity
      # равенство по идентичности, а не по значению
      def ==(other)
        other.is_a?(self.class) && other.id == id && !id.nil?
      end
      alias_method :eql?, :==

      def hash = [self.class, id].hash
    end
  end
end

Тактический паттерн · 03

Aggregate Root — страж инвариантов

Агрегат — это кластер объектов, который изменяется как единое целое. У него один корень (Aggregate Root) — единственная точка входа. Внешний код не лезет внутрь напрямую, а вызывает методы корня, который гарантирует инварианты: правила, всегда истинные для агрегата (например, «нельзя поставить на паузу отменённую подписку»).

Состояние меняется только через осмысленные доменные методы — activate, pause, cancel — а не через сеттеры. Каждый метод проверяет, что переход допустим, и фиксирует доменное событие.

domain/entities/subscription.rb
module Subscriptions
  module Domain
    class Subscription
      include Entity

      attr_reader :id, :customer_id, :plan, :price,
                  :status, :period, :events

      def initialize(id:, customer_id:, plan:, price:, status:, period:)
        @id          = id
        @customer_id = customer_id
        @plan        = plan
        @price       = price        # Money
        @status      = status       # SubscriptionStatus
        @period      = period       # BillingPeriod
        @events      = []
      end

      # --- Фабрика: создание нового агрегата ---
      def self.subscribe(customer_id:, plan:, price:, unit:, today:)
        period = BillingPeriod.starting(on: today, unit: unit)
        sub = new(
          id: nil, customer_id: customer_id, plan: plan,
          price: price, status: SubscriptionStatus.active, period: period
        )
        sub.record(Events::SubscriptionActivated.new(
          customer_id: customer_id, plan: plan, price: price
        ))
        sub
      end

      # --- Доменное поведение, а не сеттеры ---
      def pause(on:)
        ensure!(:active, "можно ставить на паузу только активную")
        @status = SubscriptionStatus.paused
        record(Events::SubscriptionPaused.new(subscription_id: id, on: on))
      end

      def resume
        ensure!(:paused, "возобновить можно только из паузы")
        @status = SubscriptionStatus.active
      end

      # Отмена раньше срока → частичный возврат за неиспользованные дни
      def cancel(on:)
        raise DomainError, "уже отменена" if status.cancelled?
        refund = refund_for(on)
        @status = SubscriptionStatus.cancelled
        record(Events::SubscriptionCancelled.new(
          subscription_id: id, on: on, refund: refund
        ))
        refund
      end

      def record(event) = events << event
      def pull_events! = events.dup.tap { events.clear }

      private

      # Инвариант перехода состояний
      def ensure!(state, msg)
        raise DomainError, msg unless status.public_send("#{state}?")
      end

      # Бизнес-правило возврата живёт в домене, не в контроллере
      def refund_for(date)
        return Money.new(amount_cents: 0, currency: price.currency) unless period.covers?(date)
        ratio = period.remaining_days_from(date).to_f / period.total_days
        price * ratio
      end
    end
  end
end
Что мы выиграли
Всё знание о том, «что значит отменить подписку», собрано в одном месте. Любой код, который меняет подписку, обязан пройти через её методы — невозможно случайно перевести отменённую подписку в активную мимо проверок.
Тактический паттерн · 04

Domain Events — факты, которые уже случились

Доменное событие — неизменяемое сообщение о том, что в домене произошло что-то значимое. Имена в прошедшем времени: SubscriptionActivated, SubscriptionCancelled. Агрегат накапливает события, а слой приложения публикует их после успешного сохранения. Так другие части системы (письма, аналитика, биллинг) реагируют, не залезая внутрь модуля.

domain/events/subscription_cancelled.rb
module Subscriptions
  module Domain
    module Events
      SubscriptionCancelled = Data.define(:subscription_id, :on, :refund) do
        def event_name = "subscriptions.cancelled"
        def occurred_at = on
      end
    end
  end
end

Минимальная шина событий — поверх встроенного в Rails ActiveSupport::Notifications, без сторонних гемов:

infrastructure/event_publisher.rb
module Subscriptions
  module Infrastructure
    module EventPublisher
      def self.publish(events)
        Array(events).each do |event|
          ActiveSupport::Notifications.instrument(
            event.event_name, event: event
          )
        end
      end
    end
  end
end

ActiveSupport::Notifications.subscribe("subscriptions.cancelled") do |*, payload|

Тактический паттерн · 05

Repository — мост между доменом и базой

Репозиторий прячет хранилище за коллекционным интерфейсом: find, save. Домен объявляет что ему нужно (интерфейс), инфраструктура решает как (ActiveRecord). Это и есть инверсия зависимостей, благодаря которой домен не зависит от Rails.

Сначала ActiveRecord-модель. Заметьте: она «тонкая» — это лишь таблица, никакой бизнес-логики. Её имя SubscriptionRecord, чтобы не путать с доменной Subscription.

infrastructure/records/subscription_record.rb
module Subscriptions
  module Infrastructure
    class SubscriptionRecord < ApplicationRecord
      self.table_name = "subscriptions"
      # колонки: customer_id, plan, price_cents, currency,
      # status, period_starts_on, period_ends_on
    end
  end
end

Реализация репозитория делает маппинг в обе стороны: строка БД → доменный агрегат и обратно. Это единственное место, где встречаются ActiveRecord и домен.

infrastructure/ar_subscription_repository.rb
module Subscriptions
  module Infrastructure
    class ArSubscriptionRepository
      NotFound = Class.new(StandardError)

      def find(id)
        to_domain(SubscriptionRecord.find(id))
      rescue ActiveRecord::RecordNotFound
        raise NotFound, "subscription #{id}"
      end

      def save(sub)
        record = sub.id ? SubscriptionRecord.find(sub.id) : SubscriptionRecord.new
        record.update!(to_columns(sub))
        # присваиваем сгенерированный id обратно в новый агрегат
        sub.instance_variable_set(:@id, record.id) if sub.id.nil?
        sub
      end

      private

      # строка БД → доменный агрегат (реконструкция, без событий)
      def to_domain(r)
        Domain::Subscription.new(
          id: r.id, customer_id: r.customer_id, plan: r.plan,
          price: Domain::Money.new(amount_cents: r.price_cents, currency: r.currency),
          status: Domain::SubscriptionStatus.new(r.status),
          period: Domain::BillingPeriod.new(
            starts_on: r.period_starts_on, ends_on: r.period_ends_on
          )
        )
      end

      # доменный агрегат → атрибуты колонок
      def to_columns(s)
        {
          customer_id: s.customer_id, plan: s.plan,
          price_cents: s.price.amount_cents, currency: s.price.currency,
          status: s.status.to_s,
          period_starts_on: s.period.starts_on,
          period_ends_on: s.period.ends_on
        }
      end
    end
  end
end
Почему два класса для одной таблицы
SubscriptionRecord — это форма хранения, Domain::Subscription — модель поведения. Их структура может расходиться: в домене price это Value Object, в базе — две колонки. Маппинг изолирует домен от схемы БД.
Чтение · важная развилка

Как читать из домена: списки, фильтры, экраны

До сих пор репозиторий умел только find(id) и save — этого достаточно, чтобы изменять подписку. Но как вывести список всех подписок клиента или таблицу для админки? Здесь новички совершают типичную ошибку: добавляют в репозиторий all и возвращают массив доменных агрегатов.

Почему «вернуть список агрегатов» — плохо
Агрегат — тяжёлый объект, защищённый инвариантами, предназначенный для изменения. Чтобы показать таблицу из 200 строк, вы реконструируете 200 агрегатов со всеми Value Objects и событиями — впустую. Вдобавок шаблону для отображения часто нужны данные из других контекстов (имя клиента, название тарифа), которых в агрегате нет, — и начинается протечка границ.

Решение — разделить команды и запросы (CQS, прагматичный родственник CQRS):

Команда (запись)Запрос (чтение)
ЦельИзменить состояниеПоказать данные
Через чтоRepository → агрегатQuery → плоский DTO
ИнвариантыЗащищены агрегатомНе нужны — read-only
Форма данныхDomain::SubscriptionSubscriptionListItem (struct)
Примерcancel(id:)list_for_customer(id)

Сторона чтения не обязана уважать богатую модель: она может бить прямо в базу, джойнить таблицы и отдавать ровно те поля, что нужны экрану. Это отдельный объект — Query (или read-model), который живёт в инфраструктуре и возвращает простые DTO, а не агрегаты.

DTO для строки списка

Плоский, неизменяемый объект чтения — только данные для отображения, никакого поведения:

application/read_models/subscription_list_item.rb
module Subscriptions
  module ReadModels
    # DTO для экрана: ровно то, что нужно таблице, и ничего лишнего
    SubscriptionListItem = Data.define(
      :id, :plan, :status, :price, :renews_on
    ) do
      def active? = status == "active"
    end
  end
end

Query-объект

Здесь разрешено пользоваться всей мощью ActiveRecord: where, order, джойны, пагинация, pluck. Цель — отдать готовые DTO быстро и без реконструкции агрегатов.

infrastructure/queries/subscription_queries.rb
module Subscriptions
  module Infrastructure
    class SubscriptionQueries
      # список подписок клиента — read-only, без агрегатов
      def list_for_customer(customer_id, status: nil)
        scope = SubscriptionRecord.where(customer_id: customer_id)
        scope = scope.where(status: status) if status
        scope.order(period_starts_on: :desc).map { |r| to_item(r) }
      end

      # пагинированный список для админки
      def page(page: 1, per: 25)
        SubscriptionRecord
          .order(id: :desc).limit(per).offset((page - 1) * per)
          .map { |r| to_item(r) }
      end

      private

      def to_item(r)
        ReadModels::SubscriptionListItem.new(
          id: r.id, plan: r.plan, status: r.status,
          price: format("%.2f %s", r.price_cents / 100.0, r.currency),
          renews_on: r.period_ends_on
        )
      end
    end
  end
end
Нужны поля из других контекстов?
Если таблице нужно имя клиента или красивое название тарифа, query-объект может сделать joins к таблицам других модулей на стороне чтения и положить результат в DTO. Это не нарушает границы домена, потому что запись по-прежнему идёт только через агрегат и фасад — протекает лишь read-only проекция.

Открываем чтение через фасад

Query доступен снаружи так же, как команды — через тот же публичный API модуля. Контроллер списка остаётся тонким:

subscriptions.rb — добавляем чтение
module Subscriptions
  module_function

  # --- запросы (чтение) ---
  def list_for_customer(customer_id, status: nil)
    queries.list_for_customer(customer_id, status: status)
  end

  def queries = Infrastructure::SubscriptionQueries.new
  private_class_method :queries
end

# контроллер — тонкий, получает готовые DTO
class SubscriptionsController < ApplicationController
  def index
    @subscriptions = Subscriptions.list_for_customer(
      current_user.id, status: params[:status]
    )
    # во вью: @subscriptions.each { |s| s.plan, s.price, s.status ... }
  end
end
Когда чтение можно упростить
Если приложение небольшое и проекции совпадают со схемой, отдельный DTO-слой избыточен — можно вернуть из query голые хэши через pluck или даже сами SubscriptionRecord в режиме read-only. Главное правило неизменно: чтение не проходит через агрегат, а запись — только через него. Полноценный CQRS с отдельной БД для чтения нужен лишь при высокой нагрузке; здесь достаточно разделения на уровне объектов.

Application Service — сценарий использования

Доменные объекты не знают про транзакции и репозитории. Их связывает Application Service (use-case): он загружает агрегат, вызывает доменный метод, сохраняет результат в транзакции и публикует накопленные события. Один класс — один сценарий.

application/cancel_subscription.rb
module Subscriptions
  module Application
    class CancelSubscription
      def initialize(repository:, publisher: Infrastructure::EventPublisher)
        @repository = repository
        @publisher  = publisher
      end

      # возвращает сумму возврата (Money)
      def call(subscription_id:, on: Date.current)
        refund = nil

        ActiveRecord::Base.transaction do
          subscription = @repository.find(subscription_id)
          refund = subscription.cancel(on: on)   # вся логика — в домене
          @repository.save(subscription)
          @events = subscription.pull_events!
        end

        # публикуем ТОЛЬКО после успешного коммита
        @publisher.publish(@events)
        refund
      end
    end
  end
end
Порядок важен
Сначала транзакция и коммит, потом публикация событий. Иначе письмо о возврате уйдёт, а транзакция откатится — клиент получит подтверждение несостоявшейся отмены.
Сборка

Полный сценарий: от контроллера до базы

Соберём всё вместе. Запрос отмены подписки проходит сверху вниз и обратно. Заметьте: контроллер не знает ни о домене, ни о репозитории — он говорит только с фасадом.

путь запроса
# 1. Контроллер — тонкий, знает только про фасад модуля
class SubscriptionsController < ApplicationController
  def destroy
    refund = Subscriptions.cancel(subscription_id: params[:id])
    render json: { refund: refund.to_s }
  rescue Subscriptions::Domain::DomainError => e
    render json: { error: e.message }, status: :unprocessable_entity
  end
end

# 2. Фасад собирает зависимости и делегирует use-case
# 3. CancelSubscription грузит агрегат через репозиторий
# 4. subscription.cancel(on:) применяет бизнес-правила в домене
# 5. Репозиторий мапит агрегат обратно в строку и сохраняет
# 6. После коммита публикуются события → письмо о возврате
Граница контекста

Публичный API модуля — единственная дверь

Снаружи модуль должен выглядеть как чёрный ящик с парой методов. Все внутренние классы — детали реализации. Фасад собирает зависимости (composition root для модуля) и предоставляет осмысленный интерфейс.

subscriptions.rb — фасад
module Subscriptions
  module_function

  def subscribe(customer_id:, plan:, price_cents:, unit:)
    price = Domain::Money.new(amount_cents: price_cents)
    Application::ActivateSubscription.new(repository: repository)
      .call(customer_id: customer_id, plan: plan, price: price, unit: unit)
  end

  def cancel(subscription_id:, on: Date.current)
    Application::CancelSubscription.new(repository: repository)
      .call(subscription_id: subscription_id, on: on)
  end

  def repository = Infrastructure::ArSubscriptionRepository.new
  private_class_method :repository
end
Граница, которую можно усилить
Хотите запретить остальному приложению обращаться к Subscriptions::Domain напрямую — добавьте Packwerk (это инструмент Shopify, но не зависимость кода). Без него граница держится дисциплиной команды и код-ревью: правило «только через фасад».

Тестирование: домен без базы данных

Главная награда за разделение слоёв — доменные правила тестируются как чистые Ruby-объекты, мгновенно, без подключения к БД и без фабрик ActiveRecord.

spec/subscriptions/subscription_spec.rb
RSpec.describe Subscriptions::Domain::Subscription do
  def build_active(price_cents: 3000)
    described_class.subscribe(
      customer_id: 1, plan: "pro",
      price: Subscriptions::Domain::Money.new(amount_cents: price_cents),
      unit: :monthly, today: Date.new(2026, 1, 1)
    )
  end

  it "запрещает паузу для отменённой подписки" do
    sub = build_active
    sub.cancel(on: Date.new(2026, 1, 10))
    expect { sub.pause(on: Date.current) }
      .to raise_error(Subscriptions::Domain::DomainError)
  end

  it "считает частичный возврат за неиспользованные дни" do
    sub = build_active(price_cents: 3100)        # 31 день в январе
    refund = sub.cancel(on: Date.new(2026, 1, 21))  # 10 дней остаётся
    expect(refund.amount_cents).to eq(1000)
  end

  it "фиксирует событие при активации" do
    sub = build_active
    expect(sub.events.map(&:event_name)).to include("subscriptions.activated")
  end
end

Application Service тестируют с in-memory репозиторием — тот же интерфейс, хранение в хэше. Интеграционные тесты против настоящего ArSubscriptionRepository и базы пишут отдельно и реже.

spec/support/in_memory_repository.rb
class InMemorySubscriptionRepository
  def initialize = (@store = {}; @seq = 0)

  def find(id) = @store.fetch(id)

  def save(sub)
    id = sub.id || (@seq += 1)
    sub.instance_variable_set(:@id, id)
    @store[id] = sub
  end
end

Резюме
DDD на Rails — это не борьба с фреймворком, а дисциплина границ. Чистый домен на pure Ruby выражает правила, репозиторий изолирует базу, application service оркеструет, фасад прячет внутренности. Никаких внешних гемов: Data.define, ActiveSupport::Notifications и Zeitwerk дают всё необходимое.

Применяйте это выборочно. Подписки, биллинг, расчёт цен, рабочие процессы заказа — стоят слоёв. Справочник стран или CRUD-админка — нет. Сила DDD в том, чтобы тратить сложность там, где она окупается.
