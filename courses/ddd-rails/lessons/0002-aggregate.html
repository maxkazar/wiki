Агрегат и корень агрегата

Модуль с одной дверью: как собрать Entity и Value Objects в единицу, которая сама охраняет свои правила — и почему снаружи у неё ровно одна точка входа.

В Лессоне 01 мы собрали мелкий кирпич — Value Object. Теперь — несущая конструкция твоего командного стандарта: Aggregate (агрегат). Это и есть «модуль с одной точкой входа», ради которого ты учишь DDD. Tangible win этого лессона: научиться находить корень агрегата и проводить через него все изменения, чтобы инварианты нельзя было нарушить.

Идея за одну минуту

Эрик Эванс: «Агрегат — это кластер связанных объектов, который мы рассматриваем как единое целое с точки зрения изменения данных».1 Три слова здесь несущие:

  • Aggregate — группа Entity и Value Objects, которые меняются вместе, в одной транзакции. Это граница консистентности.
  • Aggregate Root (корень) — главная Entity группы и единственная дверь. Снаружи можно держать ссылку только на корень (по его id); внутренние объекты спрятаны.1
  • Invariant (инвариант) — правило домена, которое всегда должно быть истинно. Корень проверяет его при каждой команде: приведёт ли изменение к валидному состоянию? Если нет — отклоняет целиком.1
Метафора двери Агрегат — это здание с одной проходной (корень). Хочешь что-то изменить внутри — идёшь через проходную, где висят правила (инварианты). Нельзя залезть в окно и переставить мебель в обход охраны. В коде «залезть в окно» = дёрнуть внутреннюю Entity напрямую.

Наш агрегат: Топология ИС

Топология — граф: Узел (Node) и Связь (Link) между узлами. Что обязано быть истинным всегда — это инварианты графа:

  • Нет висячих связей: Link соединяет только узлы, которые есть в этой топологии.
  • Нет связи узла с самим собой (по правилам нашего домена).
  • Нельзя удалить узел, пока на него ссылаются связи.

Кто гарантирует эти правила? Если «каждый контроллер сам» — гарантий нет. В DDD их гарантирует корень — Topology.

⟂ Было: has_many наружу, инварианты держать некому

# Топология как «мешок» AR-моделей с открытыми ассоциациями
class Topology < ApplicationRecord
  has_many :nodes
  has_many :links
end

# Контроллер лезет внутрь напрямую — «через окно»:
topology.links.create!(from_id: 7, to_id: 999)
# to_id=999 нет в этой топологии → висячая связь.
# Граф порвался. Никто не проверил инвариант.

Соберём Topology как корень агрегата: единственная дверь с методами-командами, которые проверяют инварианты до изменения.

✓ Стало: одна дверь, инварианты внутри корня

# app/domain/topology/topology.rb
module Topology
  class Graph < ApplicationRecord
    self.table_name = "topologies"
    has_many :nodes, dependent: :destroy   # внутри агрегата — ок
    has_many :links, dependent: :destroy

    # КОМАНДА: добавить связь. Дверь, а не сеттер.
    def connect(from_node, to_node)
      ensure_inside!(from_node, to_node)        # инвариант 1
      raise SelfLink if from_node == to_node     # инвариант 2
      links.create!(from: from_node, to: to_node)
    end

    def remove_node(node)
      raise NodeInUse if links.exists?(          # инвариант 3
        "from_id = :id OR to_id = :id", id: node.id)
      node.destroy!
    end

    private

    def ensure_inside!(*ns)
      raise NodeOutside unless ns.all? { _1.topology_id == id }
    end
  end
end

# Контроллер теперь говорит с корнем, а не с потрохами:
topology.connect(node_a, node_b)   # инварианты гарантированы
Корень как транзакция Команда корня = одна транзакция, оставляющая агрегат в валидном состоянии. connect либо создаёт корректную связь, либо падает — промежуточного «полу-битого» графа не бывает. Это и есть «граница консистентности».

Граница агрегата: ссылки по id, не по ассоциации

А что с Находкой (Finding) — уязвимостью, найденной на узле? Соблазн — belongs_to :node. Но Finding и Topology — разные агрегаты. Источник прямо предупреждает: Rails-ассоциации belongs_to/has_many через границу агрегата провоцируют лишнюю связанность и подгрузку чужого агрегата.1

✗ Через границу — ассоциацией
class Finding < AR
  belongs_to :node
end

Тянет за собой весь агрегат топологии, связывает жизненные циклы.

✓ Через границу — по id
class Finding < AR
  # node_id:integer
  # просто идентичность
end

Finding ссылается на узел по id. Агрегаты независимы.

Правило: внутри агрегата — обычные ассоциации; через границу — только id корня другого агрегата.

Держи агрегат маленьким «Правильный размер агрегата — это нужный размер границы консистентности».1 Не заси́сывай в один агрегат всё подряд: чем он больше, тем длиннее транзакции и блокировки. Если два набора данных не обязаны меняться вместе в одной транзакции — это два агрегата.

Проверь себя

По памяти, не подглядывая. Усилие припоминания — то, что переводит знание в долгую память.

Контроллеру нужно добавить связь в граф. Как правильно?

Через команду корня. Только корень проверяет инварианты (оба узла внутри, нет петли). Прямое создание Link — это «влезть в окно» в обход охраны: инварианты никто не гарантирует.

Finding (уязвимость) ссылается на Node из агрегата топологии. Как?

По идентичности (node_id). Finding и Topology — разные агрегаты. Через границу держим только ссылку-id на корень/сущность другого агрегата, а не AR-ассоциацию, которая тянет и связывает чужой агрегат целиком.

Что в первую очередь определяет границу агрегата?

Граница консистентности. Агрегат — это набор объектов, которые должны оставаться согласованными вместе, в одной транзакции. Не экран, не таблица — а инварианты, которые надо удерживать атомарно. Поэтому агрегаты держат маленькими.

Как защитить на ревью Два вопроса к PR. (1) «Где здесь инвариант и кто его охраняет?» — если изменение внутреннего объекта идёт в обход корня (topology.links.create! из контроллера), проси завести команду на корне. (2) «Эта ассоциация пересекает границу агрегата?» — если belongs_to тянет другой агрегат, предлагай ссылку по id. Формула для команды: «у каждого агрегата одна дверь; через границу — только id».
