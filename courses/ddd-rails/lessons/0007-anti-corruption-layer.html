Анти-коррупционный слой

Граница с внешним миром. Как формат стороннего сканера переводится в твой домен — и не протекает в него. Прямое продолжение этапа 5 капстоуна.

В капстоуне всплыл вопрос: куда осесть сырому формату внешнего сканера, чтобы он не загрязнил домен? Ответ — Анти-коррупционный слой (Anti-Corruption Layer, ACL). Tangible win: научиться защищать язык своего домена от чужой модели на границе интеграции.

Идея за одну минуту

Эрик Эванс (паттерн описан в Domain-Driven Design): размести между подсистемами с разной семантикой фасад/адаптер, который переводит запросы из одной модели в другую.1 Зачем — Microsoft формулирует прямо: внешние системы часто несут «запутанные схемы данных и устаревшие API»; если новый код вынужден подстраиваться под их семантику, это «разлагает то, что иначе было бы чисто спроектированным приложением».1

Главная мысль Без ACL чужой словарь (поля Nessus, шкала OpenVAS, формат Qualys) расползается по твоему домену: модель Finding обрастает plugin_id, risk_factor и прочими «не своими» понятиями. ACL — это шлюз, где чужой язык переводится в твой ровно один раз, и дальше домен говорит только на своём.

Три слоя ACL

Канонически ACL разбит на три части — каждая PORO:2

Внешний сканер
Nessus / OpenVAS
{plugin_id, risk_factor: "Critical", cvss3_base}
→
Adapter
HTTP, авторизация, ретраи
NessusAdapter
→
Translator
чужой формат → домен
NessusTranslator
→
Домен
чистые объекты
Finding(CveId, CvssScore)
  • Adapter — протокол: HTTP-вызовы, авторизация, ретраи, ошибки. Отдаёт сырой ответ.2
  • Translator — перевод: сырой JSON → доменные Finding/CvssScore/CveId. Здесь умирает чужой словарь.
  • Facade (порт) — публичный вход в языке твоего домена: VulnerabilityScanner#scan(asset_id) → [Finding].

Как это выглядит в Rails

# Порт в языке домена — от него зависит RunScan, не от Nessus
module Vulnerabilities
  class VulnerabilityScanner          # Facade / порт
    def initialize(adapter:, translator:)
      @adapter, @translator = adapter, translator
    end
    def scan(asset_id)
      raw = @adapter.fetch(asset_id)   # чужой формат
      @translator.to_findings(raw)     # → доменные Finding
    end
  end
end

# Translator — единственное место, где живёт чужой словарь
class NessusTranslator
  def to_findings(raw)
    raw.fetch("vulnerabilities").map do |v|
      Vulnerabilities::Finding.new(
        cve:  Vulnerabilities::CveId.new(v.fetch("cve")),
        cvss: Vulnerabilities::CvssScore.new(v.fetch("cvss3_base")),
        asset_id: v.fetch("asset_id"))
      # их "risk_factor":"Critical" НЕ берём — severity выводим из CVSS сами
    end
  end
end

Поменяли Nessus на OpenVAS — пишешь новый OpenVasAdapter + OpenVasTranslator за тем же портом. Домен и RunScan не трогаются вообще. Вот что покупает ACL.

Граница доверия = граница безопасности Внешний сканер — недоверенный ввод. Microsoft: на этой границе «применяй валидацию и санитизацию ввода», ведь подсистемы имеют разный уровень доверия.1 Для твоей ИБ-платформы это не абстракция: Translator — естественное место проверить, что CVE-id настоящий, CVSS в диапазоне, asset_id known. CvssScore.new уже валиден по построению (Лессон 01) — VO работают на тебя и здесь.
Калибровка — только перевод, без правил В ACL кладут только трансляцию. Microsoft прямо: «сосредоточь слой на логике перевода; не размещай в нём бизнес-правила или оркестрацию».1 И не строй ACL, если семантического разрыва нет: когда чужой формат почти совпадает с твоим — хватит простого маппинга. ACL окупается там, где словари реально расходятся (а у сканеров они расходятся сильно).

Проверь себя

По памяти — это решения на границе любой внешней интеграции.

Зачем вообще нужен анти-коррупционный слой на интеграции?

Чтобы чужая модель не протекла в домен. Без ACL словарь сканера (plugin_id, risk_factor) расползается по твоим моделям и «разлагает» чистый дизайн. ACL переводит чужой язык в твой ровно один раз, на границе.

В какой части ACL живёт перевод чужого формата в Finding?

В транслейторе. Adapter отвечает только за протокол (HTTP, авторизация, ретраи), Translator — за маппинг чужого JSON в доменные объекты. Домен про чужой формат не знает вовсе.

Что НЕ должно жить внутри анти-коррупционного слоя?

Бизнес-правила и оркестрация. ACL — только перевод (+ валидация ввода на границе доверия). Логика домена остаётся на агрегатах и сервисах; смешивать её с трансляцией — значит снова всё запутать.

Как защитить на ревью Три вопроса к PR с внешней интеграцией. (1) «Чужие поля (plugin_id, risk_factor) видны где-то за транслейтором?» — если да, словарь протёк, верни перевод на границу. (2) «Домен зависит от порта в своём языке или от класса сканера?» — зависимость должна смотреть на VulnerabilityScanner, не на Nessus*. (3) «Внешний ввод провалидирован на границе?» Формула: «чужой язык умирает в транслейторе; домен зависит от порта, а не от вендора».

Если у вас уже есть интеграция со сканером — принеси кусок маппинга в чат. Посмотрим, не протекают ли чужие поля в домен и где провести границу транслейтора. Это самый частый источник «гнили» в ИБ-платформах.
