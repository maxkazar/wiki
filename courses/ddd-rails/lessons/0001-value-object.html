Value Object

Первый кирпич доменного модуля: как превратить «голую колонку» в понятие домена, у которого нельзя быть невалидным.

Твоя миссия — командный стандарт модулей поверх ActiveRecord. Прежде чем строить агрегаты и доменные сервисы, нужен самый мелкий кирпич, из которого они складываются. Это Value Object (объект-значение). Он даёт первый ощутимый выигрыш: невалидное состояние перестаёт компилироваться.

Идея за одну минуту

DDD делит доменные объекты на два вида по одному вопросу: важна ли индивидуальность?

  • Entity (сущность) — у неё есть идентичность. Актив id=42 остаётся тем же активом, даже если у него сменился IP и имя. Сравниваем по id.
  • Value Object — идентичности нет, важно только значение. CVSS 7.5 — это просто 7.5; две «семь-с-половиной» взаимозаменяемы. Сравниваем по содержимому.1
Ключевой признак Если объект можно целиком заменить другим с тем же значением и ничего не сломается — это Value Object. Купюру в 100 ₽ ты меняешь на другую сотню не глядя на серийный номер. CVSS-оценку, IP-адрес, CVE-идентификатор — так же.

У Value Object три свойства, и каждое — рычаг против хаоса в коде:

  1. Иммутабельность. Создан — не меняется. Нужно другое значение — создаёшь новый объект. Нет «кто-то поменял оценку под нами».
  2. Валидность по построению. Объект нельзя создать в невалидном состоянии. Нет CVSS = 11.0 и нет severity, не совпадающего с оценкой.
  3. Поведение едет вместе с данными. «Какая это severity?», «это критично?» — методы самого объекта, а не разбросанные по хелперам if-ы.2

Как это выглядит у нас (домен ИБ)

Типичная уязвимость в нашей платформе: ActiveRecord-модель Finding с колонками cvss_score:float и severity:string. Rails-привычка — держать оба поля на модели и считать severity где-нибудь рядом:

⟂ Было: данные и логика размазаны по AR

# app/models/finding.rb
class Finding < ApplicationRecord
  validates :cvss_score, inclusion: { in: 0.0..10.0 }
  validates :severity, inclusion: { in: %w[none low medium high critical] }

  # severity и score можно рассинхронить — никто не мешает
  # записать score=9.0, severity="low". База это проглотит.
end

# app/helpers/findings_helper.rb — логика «уехала» от данных
def severity_badge(finding)
  case finding.cvss_score
  when 9.0..10.0 then "critical"
  when 7.0...9.0 then "high"
  # ...тот же case продублирован в трёх местах проекта
  end
end

Доменное понятие здесь — не «два поля», а одно: CVSS-оценка. У неё есть значение (0.0–10.0) и поведение (в какой severity-корзине она лежит). Соберём её в Value Object — PORO, без наследования от ActiveRecord::Base:

✓ Стало: одно понятие домена, невалидность непредставима

# app/domain/security/cvss_score.rb
module Security
  class CvssScore
    include Comparable
    SEVERITIES = { 0.0..0.1 => :none, 0.1..4.0 => :low,
                   4.0..7.0 => :medium, 7.0..9.0 => :high,
                   9.0..10.0 => :critical }.freeze

    attr_reader :value

    def initialize(value)
      value = Float(value)
      unless (0.0..10.0).cover?(value)
        raise ArgumentError, "CVSS вне диапазона: #{value}"
      end
      @value = value
      freeze                      # иммутабельность
    end

    def severity                  # поведение едет с данными
      SEVERITIES.find { |range, _| range.cover?(value) }.last
    end

    def critical? = severity == :critical

    def <=>(other) = value <=> other.value   # сравнение по значению
    def ==(other) = other.is_a?(CvssScore) && value == other.value
    def to_s = format("%.1f", value)
  end
end

Теперь severity невозможно рассинхронить со score — её больше нет как отдельной колонки, она вычисляется из единственного источника правды. А AR-модель хранит примитив и отдаёт Value Object:

# app/models/finding.rb
class Finding < ApplicationRecord
  def cvss = Security::CvssScore.new(cvss_score)
end

finding.cvss.severity   # => :high
finding.cvss.critical?  # => false
Security::CvssScore.new(11.0)  # => ArgumentError, до базы не дойдёт
Почему PORO, а не AR Value Object не имеет идентичности и жизненного цикла в базе — наследовать его от ActiveRecord::Base незачем. Это обычный Ruby-объект.2 Под капотом можно использовать и composed_of Rails, но начни с чистого PORO — стандарт команды должен быть переносимым и явным.

Проверь себя

Не подглядывая выше — вспомни по памяти. Это и есть тренировка, которая закрепляет надолго.

IP-адрес актива в топологии. Entity или Value Object?

Value Object. Адрес 10.0.0.5 взаимозаменяем с любым другим 10.0.0.5 — индивидуальности нет, важно только значение. Сам актив — Entity, но его адрес — значение.

Зачем делать Value Object иммутабельным (freeze)?

Чтобы значение нельзя было тихо подменить. Если один кусок кода держит ссылку на CVSS-оценку, а другой её мутирует — получаем «оценка поменялась под нами». Иммутабельность убирает целый класс багов: нужно другое значение — создаёшь новый объект.

Главный выигрыш переноса severity внутрь CvssScore?

Невалидное состояние стало непредставимым. Раньше score и severity жили в двух колонках и могли противоречить друг другу. Теперь severity вычисляется из единственного источника правды — рассинхрон физически невозможен. Это и есть «make illegal states unrepresentable».

Как защитить на ревью Когда коллега принесёт PR с парой колонок вроде score + level, ip + subnet_mask, cve_id-строкой с ручной валидацией — спроси: «это одно понятие домена? могут ли эти поля противоречить друг другу?» Если да — это кандидат в Value Object. Формулировка для команды: «одно понятие — один объект; невалидное состояние не должно компилироваться».
