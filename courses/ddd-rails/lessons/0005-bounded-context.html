Ограниченный контекст

Сам модуль. Где провести границу между Топологией, Уязвимостями и Документацией — и почему «Актив» в каждой из них это разная модель, а не дублирование.

До сих пор мы делали кирпичи: Value Object, Агрегат, Доменный сервис. Теперь — стены. Ограниченный контекст (Bounded Context) и есть тот модуль, ради которого ты учишь DDD. Tangible win: научиться проводить границу модуля по смыслу, а не по папкам.

Идея за одну минуту

Мартин Фаулер: модель работает как единый язык общения, поэтому «нужна другая модель, когда меняется язык».1 Граница контекста проходит там, где смысл слова сдвигается. И ключевое признание DDD: «полная унификация модели предметной области для большой системы не осуществима и не окупается».1

Главная мысль Ограниченный контекст — это граница, внутри которой термин имеет ровно одно значение. Снаружи то же слово может значить другое — и это нормально. Не пытайся сделать одну модель «Актив» на всю систему: это та самая унификация, которая не окупается.

«Актив» — одно слово, три модели

В твоей платформе слово Актив живёт в трёх подсистемах. Соблазн новичка — одна гигантская модель Asset с полями на все случаи. Но присмотрись: это разные понятия, у которых случайно одно имя.1

Топология

Актив = узел графа: позиция в сети, интерфейсы, порты, связи.

Node(ip, ports, links)

Уязвимости

Актив = цель скана: находки, экспозиция, риск.

ScanTarget(findings, risk)

Документация

Актив = запись в отчёте: владелец, описание, комплаенс-статус.

AssetEntry(owner, status)

↑ связаны не наследованием, а общей идентичностью: asset_id ↑

Три модели, заточенные каждая под свой язык, связаны общим id. Узлу топологии не нужны поля комплаенса; записи в отчёте не нужны порты. Одна «бог-модель» на всех = поля-сироты, условные ветки «а в этом случае не заполнять» и вечные конфликты на ревью.

Как это выразить в Rails (без engines)

Тебе не нужны Rails Engines или Packwerk, чтобы провести границу. Достаточно пространства имён и дисциплины. Контекст = модуль-неймспейс со своими агрегатами и сервисами внутри:

app/domain/
  topology/         # контекст «Топология»
    graph.rb            Topology::Graph   (корень агрегата)
    node.rb             Topology::Node
  vulnerabilities/  # контекст «Уязвимости»
    scan.rb             Vulnerabilities::Scan
    finding.rb          Vulnerabilities::Finding
    cvss_score.rb       Vulnerabilities::CvssScore
  documentation/    # контекст «Документация»
    document.rb         Documentation::Document
    asset_entry.rb      Documentation::AssetEntry

Через границу — по id и через интерфейс

Контексты не лазают друг другу в потроха. Reevoo (DDD-монолит на Rails) формулирует жёстко: «доменный модуль не может напрямую вызывать код других модулей».2 Связь — по идентичности и через публичный вход:

✗ Контекст лезет внутрь
Documentation::Report
 .where(node: Topology::Node.find(x))

Документация знает внутренности топологии. Изменишь Node — сломаешь отчёты.

✓ Через id и фасад
Topology.summary_for(
  asset_id) # публичный
  # интерфейс контекста

Документация просит у топологии данные через одну дверь. Внутренности скрыты.

Способы перехода границы, от простого к строгому: (1) ссылка по asset_id; (2) публичный метод-фасад контекста (Topology.summary_for(id)); (3) доменные события (ScanCompleted → документация подписана) — для слабой связанности.2

Куда это масштабируется Reevoo доводят изоляцию до отдельных engines и схем БД на контекст2 — это твой out of scope сейчас. Но правила те же на любом масштабе: id вместо ассоциаций, публичный интерфейс вместо лазания внутрь. Начни с неймспейсов + дисциплины; механизм усилишь позже, когда границы устоятся.
Калибровка — не дроби лишнего Контекст — это реальная граница языка, а не «каждой модели по контексту». Если в двух местах «Актив» значит одно и то же — это один контекст. Дроби там, где слово меняет смысл или где разные команды/процессы владеют разными правилами. Слишком много мелких контекстов = та же связанность, только с церемонией.

Проверь себя

По памяти — это решения уровня архитектуры модуля, их ты будешь принимать на дизайн-ревью.

Где правильнее всего провести границу ограниченного контекста?

Где меняется смысл термина. Модель — это язык; нужна другая модель, когда язык меняется (Фаулер). «Актив» в топологии и в документации — разные понятия → разные контексты. Папки и таблицы вторичны.

«Актив» нужен и топологии, и документации. Как моделировать?

Две модели, общий id. Каждый контекст держит свою модель «Актива», заточенную под свой язык; связь — по идентичности. Единая бог-модель = унификация, которая «не окупается» (Фаулер).

Документации нужны данные узла из топологии. Правильный путь?

Публичный фасад по id. Контекст не лезет в чужие внутренности — «модуль не может напрямую вызывать код других модулей». Просим данные через одну дверь (или подписываемся на событие), внутренности топологии скрыты.

Как защитить на ревью Два вопроса к дизайн-PR. (1) «Этот термин здесь значит то же, что и в соседнем модуле?» — если нет, это сигнал границы контекста, а не повод тащить одну модель. (2) «Этот модуль обращается к чужим внутренностям или через публичный вход?» — прямой OtherContext::Model.where(...) заменяй на фасад/событие + ссылку по id. Формула: «граница — по смыслу слова; через границу — id и публичный интерфейс».
Первоисточник — прочитай это Bounded Context — Martin Fowler. Канонический разбор: почему единая модель не масштабируется и как язык задаёт границы. 6 минут.

2 Rails-применение в монолите (события, изоляция, «модуль не зовёт чужой код») — Domain-driven Monolith — Reevoo Engineering.

Самый ценный разговор по этому лессону — на твоей системе: сколько у вас контекстов на самом деле? Топология/Уязвимости/Документация — это три, или, например, «Сбор топологии» и «Анализ топологии» — уже два разных? Принеси список своих сущностей в чат, и проведём границы вместе.
