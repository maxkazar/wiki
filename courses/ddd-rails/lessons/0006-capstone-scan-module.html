Спроектируй модуль «Скан → Находки»

Перевод всех пяти лессонов в один навык: спроектировать модуль с нуля и защитить каждое решение. Сначала думаешь сам — потом раскрываешь эталон и сверяешься.

Бриф В контексте «Уязвимости» нужно спроектировать ядро: запуск скана и запись находок. Скан прогоняется внешним сканером, возвращает уязвимости по активам; их надо сохранить, посчитать риск и отдать другим контекстам. Спроектируй этот модуль по DDD — как PORO/сервисы поверх ActiveRecord, без engines.

Как работать: на каждом этапе сначала набросай ответ сам (в голове или на бумаге), и только потом разворачивай «Эталон» и сверяйся по чек-листу. Попытка до подсказки — это и есть тренировка.

Контекст и язык

К какому ограниченному контексту относится модуль? Что здесь значит «Актив»? Выпиши 5–6 терминов единого языка этого модуля.

Опирайся на Лессон 05: термин живёт внутри границы.

Показать эталон и чек-лист
  • Контекст: «Уязвимости» (Vulnerabilities).
  • «Актив» здесь = ScanTarget (цель скана: адрес + находки + риск), а не узел графа из Топологии. Связь — по asset_id.
  • Язык: Скан (Scan), Находка (Finding), Уязвимость/CVE, CVSS, Severity, Статус скана (pending/running/completed/failed).

Сверься:

  • Назвал ровно один контекст, не смешал с Топологией
  • Заметил, что «Актив» здесь — другая модель, чем в Топологии
  • Термины — существительные домена, без технического жаргона

Объекты-значения

Какие здесь Value Objects? Для каждого — значение, валидность по построению и поведение. Есть ли ловушка «лишнего» VO?

Опирайся на Лессон 01 и шпаргалку.

Показать эталон и чек-лист
  • CvssScore — 0.0–10.0; валидируется в конструкторе; поведение #severity, #critical?.
  • CveId — формат CVE-YYYY-NNNN; валидируется regex в конструкторе; #year.
  • Ловушка: Severity — не отдельная хранимая колонка/VO, а поведение CvssScore. Заводить её отдельно — риск рассинхрона (Лессон 01).
  • ScanStatus — допустимо как VO-перечисление, но переходы статуса — это инварианты агрегата (этап 3), а не логика самого значения.

Сверься:

  • Каждый VO иммутабелен и валиден по построению
  • Поведение едет с данными (severity — метод, не колонка)
  • Не наплодил VO там, где хватает поведения существующего

Агрегат, корень и инварианты

Что здесь агрегат? Кто корень — Scan или Finding? Сформулируй 3 инварианта и команды корня. Какой вопрос решает, где граница?

Опирайся на Лессон 02: агрегат = граница консистентности.

Показать эталон и чек-лист

Решающий вопрос: меняются ли находки независимо от скана после создания?

  • Если нет (находки рождаются и живут вместе со сканом) → корень Scan, Finding — внутри агрегата. Они создаются в одной транзакции, граница консистентности = скан + его находки.
  • Если да (находку триажат, назначают, закрывают спустя недели, независимо от скана) → Finding «вырастает» в собственный агрегат; Scan лишь порождает их и ссылается по id.

Возьмём корень Scan (запись прогона):

  • Инварианты: (1) находку нельзя добавить в скан не в статусе running; (2) скан нельзя завершить дважды; (3) у находки валидный cvss (гарантирует VO) и непустой asset_id.
  • Команды корня: scan.record_finding(cve:, cvss:, asset_id:), scan.complete!, scan.fail!(reason).

Сверься:

  • Явно задал решающий вопрос про независимость жизненного цикла
  • Инварианты — про консистентность, и охраняет их корень
  • Изменения идут через команды-намерения, не через сеттеры

Размещение логики

Разложи четыре поведения по лестнице: (а) «находка критична?» (б) «записать находку в скан» (в) «сводка риска по скану: счётчики по severity» (г) «прогнать скан: внешний API → сохранить → уведомить». Где ловушка?

Опирайся на Лессон 03 и тренажёр.

Показать эталон и чек-лист
  • (а) → метод объекта-значения: finding.cvss.critical?.
  • (б) → команда корня: scan.record_finding(...).
  • (в) ловушка → метод самого агрегата scan.risk_summary: это один агрегат, читающий себя. Доменный сервис тут — оверинжиниринг (ср. кейс 8 тренажёра).
  • (г) → прикладной сервис-объект RunScan: внешний API + запись + уведомление = оркестрация с I/O. Доменных правил не содержит.

Сверься:

  • «Критична?» осталась на VO, не уехала в сервис
  • Сводку риска положил на агрегат, а не в доменный сервис
  • Внешний I/O — только в прикладном сервисе, домен чист

Границы: ссылки и переход в другой контекст

Как Finding ссылается на актив из Топологии? Как Документация получит результаты скана? И где «осесть» сырому формату внешнего сканера, чтобы он не протёк в домен?

Опирайся на Лессон 05: через границу — id и публичный вход.

Показать эталон и чек-лист
  • Ссылка на актив: Finding#asset_id — по идентичности, не belongs_to Topology::Node (это чужой контекст).
  • Документация ← результаты: контекст «Уязвимости» отдаёт данные через публичный фасад (Vulnerabilities.report_for(asset_id)) или событие ScanCompleted, на которое Документация подписана. Документация не лезет в Vulnerabilities::Finding.
  • Сырой формат сканера: переводится в доменные Finding прямо в RunScan, на границе. Чужой JSON в домен не проникает (это задел на анти-коррупционный слой, Лессон 07).

Сверься:

  • Через границу контекста — только id, без AR-ассоциаций
  • Другой контекст ходит через фасад/событие, не во внутренности
  • Внешний формат не протекает дальше границы модуля

Итоговый чек-лист защиты на ревью

Собери решение целиком и прогони по восьми критериям. Это же — твой шаблон код-ревью для любого нового модуля команды.

Рубрика капстоуна (8 пунктов)
  • Контекст назван; «Актив» не спутан с Топологией
  • Value Objects иммутабельны и валидны по построению
  • Severity — поведение CVSS, а не вторая колонка
  • Корень агрегата выбран осознанно (вопрос про жизненный цикл)
  • Инварианты охраняет корень, изменения — через команды
  • Сводка риска на агрегате; доменный сервис не приплетён зря
  • Внешний I/O изолирован в прикладном сервисе
  • Через границу — id + фасад/событие, чужой формат не протекает
