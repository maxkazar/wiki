Куда положить логику?

Перемешанная практика по лессонам 01–03. 10 кейсов из платформы ИБ — на каждом одно и то же меню из пяти уровней. Решай быстро, читай фидбэк сразу.

Это навыковый лессон, не знаниевый. Кейсы намеренно перемешаны (interleaving) — именно смешивание заставляет тебя различать уровни, а не катиться по инерции. На каждом кейсе одно меню:

  1. Объект-значение — поведение про одно значение → метод на нём (Лессон 01)
  2. Команда на корне — изменение состояния одного агрегата + инвариант (Лессон 02)
  3. Доменный сервис — «ничья» логика, несколько агрегатов, без I/O (Лессон 03)
  4. Прикладной сервис-объект — оркестрация: транзакция + сайд-эффекты (Лессон 03)
  5. Запах — логика сбежала от данных, вернуть на модель (анемия)
Как проходить Сначала реши сам, потом жми. Ошибка тут полезнее угадывания — она показывает, какой уровень путается. Счёт вверху; в конце — итог.

FindingRiskService берёт одну Finding, читает её же поля cvss и asset_criticality, считает число риска и возвращает. Других агрегатов не трогает.

Запах (анемия). Логика про данные одной сущности живёт снаружи — это finding.risk_score на самой модели (или VO). Сервис тут оторвал поведение от данных.

Контроллер добавляет ребро в граф: topology.links.create!(from_id: 7, to_id: 12) — напрямую, минуя проверки «оба узла внутри» и «не петля».

Команда на корне. Это изменение одного агрегата с инвариантами графа → topology.connect(a, b). Прямой links.create! — «через окно» в обход корня.

«Поверхность атаки» = берём граничные узлы топологии и считаем критические находки на них. Чистый расчёт, читает два агрегата, ничего не пишет.

Доменный сервис. Логика тянет Topology и Findings сразу, не принадлежит ни одному → PORO без состояния и I/O (AttackSurfaceCalculator).

В трёх вьюхах повторяется case finding.cvss_score; when 9.0..10 then "critical"… — определение severity по числовой оценке.

Объект-значение. Поведение про одно значение (оценку) → метод CvssScore#severity. Дублирующийся case во вьюхах — классический сигнал «поведение хочет в VO».

GenerateComplianceReport: открывает транзакцию, собирает документ из находок, шлёт письмо стейкхолдерам и ставит джобу на архивацию.

Прикладной сервис-объект. Оркестрация с сайд-эффектами (письма, джобы, транзакция). Доменных правил не содержит — только дирижирует. Не путать с доменным сервисом.

Нужно удалить узел из топологии, но только если на него не ссылается ни одна связь — иначе граф порвётся.

Команда на корне. Инвариант графа («нельзя удалить занятый узел») охраняет корень → topology.remove_node(node).

Логика «нормализовать CIDR и проверить, входит ли IP-адрес в эту подсеть». Всё — про одно значение подсети.

Объект-значение. Подсеть — VO; «входит ли IP» — её поведение → Cidr#contains?(ip). Валидность и нормализация — в конструкторе.

TopologyStatsService принимает одну топологию и считает количество её же узлов и связей, возвращает хэш статистики.

Запах. Считает данные одного агрегата по его же полям → это topology.stats на корне. Один агрегат + чтение себя = сервис не нужен.

После скана: сопоставить находки с узлами топологии по адресу и пометить, какие узлы экспонированы. Чистая доменная логика поверх двух агрегатов.

Доменный сервис. Сопоставление живёт между Findings и Topology, не внутри одного. PORO без I/O. (А вот сохранить результат и уведомить — уже задача прикладного сервиса.)

RunScan: дёргает внешний сканер по API, сохраняет полученные находки в БД и шлёт уведомление команде безопасности.

Прикладной сервис-объект. Внешний API, запись в БД, уведомление — оркестрация с I/O. Доменных правил нет, только координация шагов.

Перенос на ревью Эти пять вопросов — твой чек-лист на любом PR с новым классом. Особенно цепляй два «запаха»: сервис, который читает поля одной сущности и кладёт назад (кейсы 1, 8) — почти всегда метод модели. Это и есть «отличить честный код от оверинжиниринга».

Какой-то кейс показался спорным или у тебя в коде есть «пограничный» класс — кидай в чат, разберём, почему он сел именно на этот уровень.
