Лессон 03 · DDD в Rails · Калибровка

Где живёт логика?

Доменный сервис vs толстая модель vs сервис-объект. Самый частый способ испортить «командный стандарт» — выносить в сервисы то, что должно жить на агрегате.

Этот лессон прямо отвечает на твою задачу: отличать честный доменный код от оверинжиниринга на ревью. У новичков в DDD один типовой рефлекс — на любую логику плодить SomethingService. Результат — анемичная модель (anemic domain model): модели превращаются в «голые держатели данных без поведения».1 Tangible win: научиться класть логику на правильный уровень с первого раза.

Установка: по умолчанию — на агрегат

Команда 37signals (авторы Rails) ставит вопрос ребром: вместо «как изолировать персистентность от доменной логики» — «а зачем?».1 Когда ORM хорошо вплавлен в язык, насильно разносить данные и поведение — лишняя церемония. Логика про один агрегат живёт на агрегате (метод/команда из Лессона 02), а не в отдельном сервисе.

Правило по умолчанию Поведение принадлежит тому, чьи данные оно меняет. «Эта связь — петля?» → метод Topology. «Эта оценка критична?» → метод CvssScore. Сервис-объект здесь — это вынести поведение прочь от данных и получить анемичную модель.

Лестница решений: четыре уровня

Прежде чем завести класс — спроси, на каком уровне логика. Иди сверху вниз, останавливайся на первом «да».

ВопросКудаПример (домен ИБ)
Логика про состояние одного агрегата? Метод/команда на агрегате topology.connect(a, b)
Связный пучок поведения на одной модели разросся? Concern / PORO-делегат Node::Reachability
Логика принадлежит нескольким агрегатам / это процесс домена, а не вещь? Доменный сервис (PORO) AttackSurfaceCalculator
Это оркестрация: транзакция + сайд-эффекты (письма, джобы)? Сервис-объект (прикладной) RunScan

Уровень 2: вплавить, не выносить (37signals)

Когда поведение всё ещё про одну модель, но раздувает её — не вынось наружу, а сгруппируй внутри через Concern или PORO-делегат. 37signals именно так держат модели богатыми: concerns и PORO инкапсулируют сложность внутри Active Record.1

# app/models/node/reachability.rb — concern, логика рядом с данными
module Node::Reachability
  extend ActiveSupport::Concern
  def reachable_from?(other) = topology.path?(other, self)
end

class Node < ApplicationRecord
  include Reachability   # модель остаётся богатой, не анемичной
end

Уровень 3: доменный сервис — для «ничьей» логики

Доменный сервис (Domain Service) нужен, когда логика — это деятельность домена, не принадлежащая ни одному агрегату. Признак: она тянет несколько агрегатов сразу. «Пересчитать поверхность атаки» смотрит и на топологию, и на находки — это не метод узла и не метод находки.

# app/domain/security/attack_surface_calculator.rb
module Security
  class AttackSurfaceCalculator        # PORO, без состояния
    def initialize(topology:, findings:)
      @topology, @findings = topology, findings
    end

    # назван глаголом-деятельностью домена, возвращает результат домена
    def call
      exposed = @topology.boundary_nodes
      exposed.sum { |n| @findings.critical_on(n.id).count }
    end
  end
end
Доменный сервис ≠ сервис-объект Доменный сервис содержит правила домена и не делает I/O (ни писем, ни джоб, ни внешних API) — чистый расчёт, легко тестируется. Сервис-объект (уровень 4, прикладной) — это оркестратор: открывает транзакцию, зовёт доменный код, потом шлёт уведомления и ставит джобы. Доменных правил в нём нет. Не смешивай эти два.
Калибровка — запах оверинжиниринга Если XxxService просто берёт поля модели, что-то считает и кладёт обратно — поведение сбежало от данных. Верни его на модель/агрегат. 37signals: «если у вас Active Record и проблемы с поддержкой кода — возможно, дело не в инструменте».1 Доменный сервис оправдан только когда логика честно ничья — про несколько агрегатов сразу.

Проверь себя

По памяти. Каждый ответ — это решение, которое ты будешь принимать на ревью.

«Является ли CVSS-оценка критической?» Где этой логике место?

Метод на CvssScore. Логика про состояние одного объекта-значения. Выносить её в сервис — значит оторвать поведение от данных и получить анемичную модель. Уровень 1 лестницы.

«Пересчитать поверхность атаки по топологии и находкам». Куда?

Доменный сервис. Логика тянет сразу два агрегата (Topology и Findings) и не принадлежит ни одному — это деятельность домена, «ничья». PORO без состояния и без I/O. Уровень 3.

Видишь PORO, который читает поля модели и кладёт результат назад. Реакция?

Вернуть на модель. Это запах: поведение сбежало от своих данных, модель анемична. Сервис оправдан, только если логика ничья (несколько агрегатов) или это прикладная оркестрация с сайд-эффектами.

Как защитить на ревью Перед каждым новым XxxService прогоняй PR по лестнице: (1) про один агрегат? → метод на нём. (2) раздувает одну модель? → concern. (3) ничья, про несколько агрегатов? → доменный сервис (PORO, без I/O). (4) оркестрация с сайд-эффектами? → прикладной сервис-объект. Формула команды: «логика — на агрегат по умолчанию; сервис — только для ничьей логики или оркестрации».
Первоисточник — прочитай это Active Record, nice and blended — 37signals. Контр-перспектива от авторов Rails: почему не стоит выносить логику в сервисы и как держать модели богатыми через concerns и PORO. Обязательно для калибровки командного стандарта. 8 минут.

Спорный случай из твоего кода — «вот этот сервис, он честный или оверинжиниринг?» — приноси в чат. Разберём по лестнице на твоём примере; это ровно тот навык, что нужен на ревью.
